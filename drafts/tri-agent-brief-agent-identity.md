# 三方收斂議題：Symbolon 要不要現在就補「agent 身分驗證」？

## 背景（共用事實，請都以此為準）
Symbolon = 一個 CC0 開放協定，用一張可驗證的 JSON「承諾收據」，記錄「AI agent 代表某人，向某商家承諾了一筆可折抵的規劃費，後來這筆承諾發生了什麼事」。已上線，Nexus(nexus.quanda.media) 是全球第一個生產簽發者。規格 v0.1 現況重點：

- **信任地基 = 商家網域**：`issuer.id` 是網址，公鑰只能放在該網域 `/.well-known/symbolon.json`，抓鑰匙禁跨網域轉址。剛做完「表頭綁定」硬化：改任何表頭欄位（含 issuer/agent 欄）→ 每個 event hash 全碎。商家簽章用 Ed25519。
- **三種角色**：issuer(商家)、agent(下單機器人)、principal(背後真人)。
  - 商家身分：綁網域+簽章，驗得出。✅
  - **agent 身分：目前 `agent.id` 只是自填字串（如 `urn:agent:claude`），無任何驗證，誰都能填。** ⚠️
  - 真人身分：`principal_ref` 刻意匿名（連電話雜湊都不放，怕暴力破解），商家私下留底。🔒（此為刻意設計，本議題不動它）
- 規格自己把「agent identity attestation (DID/VC envelopes)」列在 §5「以後再說」清單。
- 設計信條：規格要「robots.txt 大小」，太大就會死；Symbolon 只答「付款前承諾了什麼、後來如何」，不搶 VC(誰是這方)、AP2/ACP(可否付款)、x402/L402(機器怎麼付) 的地盤。

## 要收斂的決策
**「Symbolon 現在（v0.1 或緊接的 v0.1.x）就該加入某種 agent 身分綁定」的把握度，請給 0–100%。**

並具體指出你主張的做法屬於下列哪一種（或提出更好的）：
- **A. 現在就做完整版**：納入 DID/VC 信封，agent 出示可驗證憑證，簽章綁定。
- **B. 現在做輕量綁定**：agent 用自己的金鑰對 event 做「共簽 (co-sign)」，`agent.id` 綁到那把公鑰的指紋（不引入整套 DID/VC，只證明「同一個 agent 從頭到尾」+ 商家見過這把鑰匙）。
- **C. 維持現狀不動**，只把「agent.id 未驗證、勿當身分證明」這個限制在 spec 裡寫得更白。
- **D. 其他**（請描述）。

## 評估時請權衡
1. 真實威脅：不驗 agent 身分，實際會被誰、怎麼利用？（冒名？重放？農場刷規劃費？）商家簽章+承諾金額已在鏈上，agent 造假的實際獲利是什麼？
2. 「規格要小」信條 vs 補洞的張力。加東西是不是等於慢性自殺。
3. 生態相容：AP2/ACP/DID 這些鄰居協定，是不是本來就該由它們管「agent 是誰/可否付款」，Symbolon 只要留 `payment_reference.type` 這種轉接頭就好？
4. 現在只有 1 個簽發者(Nexus)、幾乎沒真流量，改協定的成本 vs 之後有量再改的成本。
5. 若選 B/A，向後相容怎麼辦（舊收據沒有 agent 簽章）。

## 輸出格式
1. 把握度 X%（單一數字）
2. 主張：A/B/C/D 之一 + 一句話
3. 3–5 點理由
4. 你最擔心的一個反對意見
